# Feature 기획

구현 시작 전에 완료해야 하는 기획 단계. 이 단계는 사용자 검토가 필요한 독립적인 페이즈이며, 기획이 승인된 후에야 [TDD 사이클](./tdd-cycle.md)로 넘어간다.

---

## 왜 기획을 분리했는가

- 기획은 **검토 대상**이다. 코드 변경과 달리 사용자와의 정렬이 필요.
- 기획 없이 바로 구현에 들어가면 잘못된 방향으로 진행되기 쉬움.
- QA 목록과 테스트 피라미드가 없으면 E2E 테스트를 선작성할 수 없음.

---

## 절차

```
[Feature 기획 시작]
├─ 1. Feature 문서 작성
├─ 2. QA 목록 작성
├─ 3. Test Pyramid 설계
└─ 4. 사용자 검토 및 승인
   ↓
[TDD 사이클로 진입] → docs/tdd-cycle.md
```

---

## 1. Feature 문서 작성

경로: `docs/features/feature-XX-<name>.md` (XX는 Phase 1 안에서의 순번)

### 필수 섹션

```markdown
# Feature XX: <이름>

## 목적

이 기능이 왜 필요한가. ground-truth.md의 두 가지 목적 중 어느 쪽에 기여하는지 명시.

## 입력

- 사용자 입력 (CLI 인자, stdin)
- 시스템 입력 (DB 상태, 파일 시스템)

## 출력

- 반환값
- 부수효과 (DB 변경, 파일 쓰기, stdout 출력)

## 성공 시나리오

정상 동작 시 입력부터 출력까지의 흐름을 서술.

## 실패 시나리오

각 에러 조건과 처리 방식:
- 입력 검증 실패
- 의존하는 상태 부재
- I/O 에러
- 기타

## 제약 조건

- 성능 (훅이면 < 50ms 등)
- 호환성 (기존 데이터 형식)
- 보안 (민감정보 처리)

## 의존하는 기능

이미 구현된 Feature 중 이 기능이 의존하는 것들. 순서를 명확히 한다.
```

## 2. QA 목록 작성

**acceptance criteria**. 각 항목은 **테스트 가능한 검증 가능 명제**여야 한다.

### 원칙

- 각 항목이 독립적으로 검증 가능
- 구체적이고 측정 가능 (애매함 금지)
- 성공 조건과 실패 조건 모두 포함

### 예시 (post-tool 훅)

```
✓ 유효한 JSON stdin 전달 시 tool_uses 테이블에 한 행 추가
✓ tool.name == "Bash"인 경우 tool.input.command가 로그에 보존
✓ duration_ms는 pre-tool 기록과의 차이로 계산
✓ session_id 필드 누락 시 "unknown" 값으로 저장
✓ 잘못된 JSON stdin은 에러 반환 + DB 미변경
✓ DB 접근 불가 시 hook-errors.log에 기록 + macOS 알림 (5분 쿨다운)
✓ 훅 실행 시간은 50ms 미만 (벤치마크)
```

### 나쁜 예 (모호함)

```
✗ 로그가 잘 저장되어야 한다
✗ 에러가 나면 적절히 처리되어야 한다
✗ 빠르게 동작해야 한다
```

## 3. Test Pyramid 설계

각 QA 항목이 어느 테스트 레벨에서 검증될지 분배.

### 레벨별 특성

| 레벨 | 대상 | 비중 | 실행 속도 |
|---|---|---|---|
| 단위 (unit) | 순수 도메인 함수, Value Object | 많음 | 빠름 (<1ms) |
| 통합 (integration) | 여러 계층 조합 (workflow + adapter) | 중간 | 빠름 (<100ms) |
| E2E | 바이너리 호출, stdin/stdout, 실제 DB 파일 | 적음 | 느림 (100ms~) |

### 분배 원칙

- **단위 테스트가 가능하면 단위로** — 피드백 속도가 가장 빠름
- **E2E는 핵심 경로만** — 전체 흐름을 한 번 검증
- **중복 회피** — 같은 로직을 여러 레벨에서 중복 테스트하지 않음

### 예시

post-tool 훅 QA 항목 분배:

| QA 항목 | 레벨 | 이유 |
|---|---|---|
| 유효한 JSON → DB 저장 | E2E | 전체 흐름 확인 |
| Bash command 보존 | 통합 | domain + adapter 조합 |
| duration_ms 계산 | 단위 | 순수 계산 로직 |
| session_id 누락 처리 | 단위 | 입력 파싱 로직 |
| 잘못된 JSON 처리 | E2E | 바이너리 레벨 에러 처리 |
| DB 접근 불가 처리 | 통합 | adapter 에러 처리 |
| 훅 실행 시간 < 50ms | 벤치마크 | 성능 테스트 (cargo bench) |

## 4. 사용자 검토 및 승인

기획 문서, QA 목록, Test Pyramid 설계가 완성되면 사용자에게 검토 요청:

- Feature 문서의 목적이 ground-truth에 부합하는가?
- QA 목록이 기능을 충분히 커버하는가?
- Test Pyramid 분배가 합리적인가?
- 제약 조건이 현실적인가?

**승인 후에만 구현 착수**. 승인 없이 구현에 들어가면 기획 자체를 다시 해야 할 수 있음.

---

## 산출물 체크리스트

기획 완료 기준:

- [ ] `docs/features/feature-XX-<name>.md` 작성됨
- [ ] 목적, 입력/출력, 시나리오, 제약 모두 명시됨
- [ ] QA 목록의 각 항목이 테스트 가능한 명제
- [ ] Test Pyramid 분배표 작성됨
- [ ] 의존하는 Feature 순서 명확
- [ ] 사용자 승인 완료

이 체크리스트가 모두 충족되면 [TDD 사이클](./tdd-cycle.md)로 넘어간다.
